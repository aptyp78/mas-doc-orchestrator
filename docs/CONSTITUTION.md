## I. Имя и миссия

**Имя проекта:** AI Canvas — суверенная среда анализа и ассистирования для первых лиц.

**Миссия:** дать руководителю C-level возможность работать с потоком входящей информации — документами, отчётами, аналитикой, перепиской — на своём языке, через естественный диалог, без посредников-аналитиков. Система самостоятельно извлекает, структурирует, связывает и хранит знания, формируя персональную федеративную среду — векторно-графовое зеркало информационного поля руководителя.

**Целевой пользователь:** CEO, CTO, CIO, глава направления — человек, принимающий решения в условиях информационной перегрузки. Не технический специалист. Не аналитик. Не оператор.

**Job-to-be-Done:** «Когда я получаю 50-страничный отчёт в 23:00, я хочу через 5 минут понимать, что в нём важно, и иметь возможность задать вопрос — без звонка аналитику».

## II. Принципы

### 1. Суверенность (Sovereignty First)

Все вычисления — локальные. Ни один байт пользовательских данных не покидает закрытый контур. Модели работают на собственном железе заказчика. Air-gap — не опция, а базовый режим.

**Почему:** C-level документы содержат стратегическую, финансовую и персональную информацию. Отправка в облако — неприемлемый риск.

### 2. Нулевое априорное знание (Zero-Shot by Default)

Система не имеет предустановленных схем документов, классификаторов доменов или шаблонов сущностей. Структура знания рождается из самих данных, а не из конфигурации.

**Почему:** руководитель работает с информацией любого типа — от финансовых моделей до военных спецификаций. Невозможно предусмотреть все форматы заранее.

### 3. Естественный интерфейс (Language-Native)

Единственный интерфейс — естественный язык. Никаких query-языков, DSL, фильтров-формочек. Руководитель спрашивает — система отвечает, ссылаясь на источники.

**Почему:** C-level не будет учить синтаксис. Если интерфейс требует инструкции — он провален. (Cognitive Load Theory: минимизация extraneous load, оптимизация germane load.)

### 4. Федеративная архитектура (Federated by Design)

Знания не сливаются в один монолитный граф. Каждый проект, направление, контур формирует собственный графовый контекст. Связи между контекстами — явные, прослеживаемые, с provenance.

**Почему:** руководитель работает в нескольких activity systems одновременно (контур ОПК, гражданский сектор, стратегия, операционная деятельность). Информация из одного контура не должна смешиваться с другим без явного указания. (Activity Theory: инструмент должен отражать структуру деятельности, а не навязывать единую.)

### 5. Доказуемость (Provenance-Grounded)

Каждое утверждение системы должно иметь traceable origin — от какого документа, какого абзаца, какого извлечения оно получено. Confidence-оценки — обязательны. Никаких «hallucination without warning».

**Почему:** цена ошибки для C-level неприемлема. Лучше «я не знаю» с confidence 0.3, чем уверенный вымысел. (Bayesian Inference: confidence-gated decision rule — P(correct | evidence) > P(error | evidence).)

### 6. Инкрементальное обучение (Continuous Learning)

Система учится на взаимодействии. Каждый вопрос руководителя, каждое уточнение, каждая поправка уточняет модель его информационного поля. Второй ответ на тот же вопрос должен быть лучше первого.

**Почему:** статическая система деградирует. Информационное поле руководителя меняется ежедневно.

## III. Методологический фундамент

Наша среда — не только для пользователя, но и для нас самих: пространство моделирования и поиска лучших решений. Методологическая база даёт конструкции легитимность за пределами «мы так решили».

### Слой 1: Продуктовая методология — как мы строим

| Фреймворк | Применение |
|-----------|------------|
| **Lean Startup** (Ries, 2011) | Build-Measure-Learn цикл: каждый прогон оркестратора → замер confidence → коррекция промптов. MVP: PDF→граф→диалог, без федерации и multi-user. Validated Learning: критерии v1 измеримы. Pivot/Persevere: порог confidence 0.85 — не достигли → пересмотр архитектуры. |
| **Jobs-to-be-Done** (Christensen, 2003) | Проектируем под «работу, которую нанимает делать систему C-level», а не под функции. Работа: «понять суть документа за 5 минут и задать вопрос». |
| **Wardley Mapping** (Wardley, 2015) | Стратегия build vs borrow: PDF-парсинг → commodity (берём PyMuPDF), оркестратор → custom (строим сами), embedding → commodity (берём готовые модели). Позволяет не переизобретать чужое. |

### Слой 2: Агентская архитектура — как думает система

| Фреймворк | Паттерн | Где у нас | Статус |
|-----------|---------|-----------|--------|
| **Anthropic Building Effective Agents** (2024) | Evaluator-Optimizer | Agent → Reflector → Agent | ✅ Реализован |
| | Orchestrator-Workers | L0 → L1 → L2 → Pass2 | 🔶 В процессе |
| | Routing | Modality Router (Text/Table/Diagram/Image) | 🔶 Спроектирован |
| | Parallelization | Параллельная обработка страниц | ⬜ Не начато |
| **Google Agentic Design Patterns** (2024) | Reflection | Reflector с self-critique | ✅ Реализован |
| | Tool Use | PyMuPDF, pdftoppm, tesseract | 🔶 В процессе |
| | Planning | Agent → разбиение на регионы → обработка | ⬜ Не начато |
| **LangGraph** (2023) | State-machine | Оркестратор как граф состояний: `PASS1→REFLECT→{done, escalate}` | ✅ Реализован |
| **Microsoft AutoGen** (2023) | Multi-agent conversation | Agent ↔ Reflector ↔ (будущий) Ensembler | 🔶 В процессе |

### Слой 3: Научные основания — почему это работает

| Фреймворк | Применение |
|-----------|------------|
| **СМД Г.П. Щедровицкого** (системно-мыследеятельностная методология) | Смысл не содержится в тексте — он возникает в деятельности. Каждый уровень конвейера (L0→L1→L2→L3→L4) — это деятельность, порождающая новый смысл. **Смысловые ходы:** трансформация смысла при переходе между уровнями. Детали: `docs/SEMANTIC_FLOW.md` |
| **Cynefin Framework** (Snowden, 2007) | Документный парсинг — Complex domain, не Complicated. В Complicated есть правильный ответ (эксперт знает). В Complex причина и следствие видны только задним числом. Наш zero-shot подход — это probe: мы не знаем, что в документе, пока не «прощупаем». Оркестратор с рефлексией — это sense→respond цикл. |
| **Activity Theory** (Engeström, 1987) | Моделирует деятельность C-level как систему: Субъект (C-level) → Инструмент (AI Canvas) → Объект (информация) → Результат (решение). Объясняет, почему федерация: у C-level несколько activity systems (контур ОПК, гражданский, стратегический), и инструмент должен отражать эту структуру. |
| **Semantic Web** (Berners-Lee, 2001) | Федеративный граф — это linked data нового поколения. Каждый узел идентифицируется (doc_id + node_id), связи типизированы (relation: contains, depends_on, implements...), данные самодостаточны в своём контексте. |
| **Bayesian Inference** | Confidence-gated routing — байесовский decision rule: P(correct \| confidence ≥ τ) > P(error \| confidence ≥ τ). Эскалация = снижение prior uncertainty через дополнительный evidence (Reflector). |
| **Cognitive Load Theory** (Sweller, 1988) | C-level уже на пределе когнитивной нагрузки. Система минимизирует extraneous load (интерфейс, навигация) и оптимизирует germane load (схемы, связи). Отсюда: естественный язык, а не query-язык; Markdown-вывод, а не JSON; confidence, а не raw probabilities. |

## IV. Архитектурные ограничения

### Аппаратные

- **Минимальный инстанс (dev):** Apple M5 Max (128 GB RAM) или эквивалент (4×GPU, 80+ GB VRAM)
- **Целевой инстанс (production):** серверный кластер с GPU (NVIDIA B-series или отечественные аналоги)
- **Сетевой режим:** air-gap capable (полностью автономная работа без интернета)
- **Модели (production):** локальные (Ollama)
- **Модели (dev):** гибрид — локальные + DashScope API + Ollama Cloud (для экспериментов)

### Программные

- **Язык:** Python 3.12+, статическая типизация (mypy)
- **ML-runtime:** Ollama (локальный инференс), DashScope API (опционально, для dev)
- **Хранение:** векторная БД (Qdrant/Milvus) + графовая БД (Neo4j/kuzu) — федеративные
- **Формат данных:** Markdown (канонический текст) + JSON-sidecar (метаданные, спаны, embeddings)
- **Интерфейс:** Open WebUI / AnythingLLM — как фронтенд, без модификаций кода

### Архитектурные

- **Оркестратор с рефлексией:** Agent → Reflector → Agent → ... до stabilisation (ADR-001)
- **5-уровневый конвейер обработки документов:** L0 OCR2 → L1 SMD-классификатор → L2 Экстракторы → L3 Онтология → L4 Рефлектор → ZoneStore. Детали: `docs/PIPELINE.md`
- **Два режима работы:** Pipeline трансформации (roles/) для загрузки документов + Интерфейс (ask_orchestrator) для запросов. Детали: `docs/adr/003-constitution-roles.md`

## V. Что мы НЕ делаем (v1)

- **Не SaaS.** Это on-premise продукт. Никакой multi-tenancy, никаких облачных аккаунтов.
- **Не поисковик.** Система не индексирует интернет. Только документы, загруженные пользователем.
- **Не чат-бот.** Диалог — средство, не цель. Цель — структурированное знание.
- **Не замена BI.** Система не строит дашборды и не выполняет SQL-запросы к базам данных.
- **Не real-time.** Документы обрабатываются асинхронно. Приоритет — точность, не скорость.
- **Не мультипользовательская (v1).** Первая версия — персональная. Федерация пользователей — позже.
- **Не web-приложение.** Фронтенд — существующий Open WebUI. Мы не пишем UI.

Границы будут уточняться по мере развития продуктовой зрелости.

## VI. Критерии готовности v1

- [ ] C-level пользователь загружает PDF (текстовый, растровый, смешанный) — система извлекает и структурирует содержание
- [ ] C-level пользователь задаёт вопрос на русском языке — система отвечает, ссылаясь на документы-источники с confidence
- [ ] Система работает полностью автономно (air-gap) на M5 Max
- [ ] Загрузка 50-страничного документа → готовность к запросам за < 5 минут
- [ ] Confidence ≥ 0.85 для текстовых документов, ≥ 0.75 для диаграмм
- [ ] Интерфейс — Open WebUI, без модификаций кода фронтенда

## VII. Процесс разработки

- **ADR перед каждым архитектурным решением** (docs/adr/)
- **Код → тесты → коммит.** Не наоборот.
- **Ветка master — всегда рабочая.** Feature-ветки для экспериментов.
- **Конфигурация отдельно от кода** (config/.env, keychain для секретов)
- **Документы-образцы в data/docs/** — репрезентативная выборка того, с чем работает руководитель
- **Методологическая рефлексия** — каждый цикл разработки сверяется с принципами и научными основаниями (раздел III)
