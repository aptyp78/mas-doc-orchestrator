# Смысловые ходы конвейера обработки документов

**Дата:** 2026-07-21
**Статус:** действующая методология
**Методологическая основа:** СМД Г.П. Щедровицкого (системно-мыследеятельностная методология)

---

## Философия смысловых ходов

**Ключевой принцип:** Смысл не содержится в тексте документа. Смысл возникает в деятельности — когда субъект применяет инструмент к объекту.

**Пример:** Аббревиатура «БИБ» не имеет смысла внутри документа. Она имеет смысл только в организационной структуре банка, которая находится вне документа. Документ — не самодостаточный источник смысла, а след деятельности, оставившей в нём указатели на внешние системы.

**Следствие для архитектуры:** Каждый уровень конвейера — это не просто обработка данных, а **деятельность**, порождающая новый смысл. Смысловой ход — это трансформация смысла при переходе от одного элемента деятельности к другому.

---

## Диаграмма смысловых ходов

```
PDF (физический объект)
  │
  │ [L0: OCR2 — извлечение примитивов]
  │ Смысл: "Документ содержит структурированные элементы"
  │
  ▼
Примитивы (текст, вектор, растр с координатами)
  │
  │ [L1: SMD-классификатор — 7 знаковых форм]
  │ Смысл: "Эти элементы образуют определённую знаковую форму"
  │
  ▼
Знаковая форма (discursive, topology, matrix, ...)
  │
  │ [L2: Экстракторы схем]
  │ Смысл: "Внутри формы есть структура (связи, иерархия, таблица)"
  │
  ▼
Схема (JSON: Venn, hierarchy, matrix, ...)
  │
  │ [L3: Онтологический маппер]
  │ Смысл: "Эти структуры образуют доменную модель (сущности + связи)"
  │
  ▼
Онтология (entities + relations + model)
  │
  │ [L4: Прагматический рефлектор]
  │ Смысл: "Из этой модели следуют риски, возможности, действия"
  │
  ▼
Рекомендации (risks, opportunities, actions)
```

## L0: PDF → Примитивы

### Вход
**PDF-страница** — физический объект, содержащий:
- Текстовые слои (скрытые или видимые)
- Векторные пути (линии, фигуры)
- Растровые изображения (фото, сканы)
- Метаданные (автор, дата, формат)

**Смысл входа:** "Это документ, но мы не знаем, что внутри"

### Преобразование
**Модуль:** `DeepSeek-OCR2-bf16-MLX` (локальный, через MCP)

**Деятельность:** Извлечение примитивов с координатами:
- Текстовые блоки → `<|ref|>text<|/ref|><|det|>[[x1, y1, x2, y2]]<|/det|>`
- Таблицы → `<|ref|>table<|/ref|><|det|>[[x1, y1, x2, y2]]<|/det|><table>...</table>`
- Изображения → `<|ref|>image<|/ref|><|det|>[[x1, y1, x2, y2]]<|/det|>`

**Производительность:** 1-3 секунды на страницу

### Выход
**Markdown с координатами** — структурированное представление страницы:
```markdown
<|ref|>title<|/ref|><|det|>[[18, 170, 504, 390]]<|/det|>
# СТРАТЕГИЯ ПРОДВИЖЕНИЯ

<|ref|>text<|/ref|><|det|>[[66, 420, 468, 491]]<|/det|>
Подготовлено для Группы ОКТО

<|ref|>table<|/ref|><|det|>[[7, 217, 999, 740]]<|/det|>
<table>...</table>
```

**Смысл выхода:** "Документ содержит структурированные элементы с пространственными координатами"

### Что теряется
- **Визуальное оформление:** шрифты, цвета, стили (кроме базовой структуры)
- **Интерактивность:** ссылки, формы, скрипты
- **Метаданные автора:** намерения, контекст создания
- **Семантические связи:** если они не выражены явно в тексте

### Что добавляется
- **Пространственная структура:** координаты каждого элемента
- **Типизация:** текст vs таблица vs изображение
- **Машиночитаемость:** готово для дальнейшей обработки
- **Прослеживаемость:** каждый элемент можно вернуть к исходным координатам

## L1: Примитивы → Знаковые формы

### Вход
**Markdown с координатами** от L0

**Смысл входа:** "Есть структурированные элементы, но мы не знаем, как они организованы"

### Преобразование
**Модуль:** `src/semiotic/classifier.py`

**Деятельность:** Классификация страницы по 7 знаковым формам (СМД Г.П. Щедровицкого):

| Форма | Описание | Критерии |
|-------|----------|----------|
| **discursive** | Линейный текст, повествование | Преобладание текстовых блоков, нет таблиц/графиков |
| **topology** | Граф, сеть, связи | Узлы + рёбра, организационная структура |
| **matrix** | Таблица, матрица | Строки + столбцы, табличные данные |
| **hierarchy** | Иерархия, дерево | Parent-child отношения, вложенность |
| **spatial** | Пространственное расположение | Карта, схема, диаграмма |
| **enumeration** | Перечисление, список | Нумерованные/маркированные пункты |
| **dynamics** | Процесс, изменение, поток | Этапы, переходы, timeline |

**Модель:** qwen3.6:35b (локальная, через Ollama)

**Промпт:** `[РОЛЬ] SMD-классификатор [ОГРАНИЧЕНИЕ] Только 7 форм, без интерпретации содержания`

### Выход
**Знаковая форма** — одна из 7 категорий:
```json
{
  "page_id": 1,
  "form": "matrix",
  "confidence": 0.85
}
```

**Смысл выхода:** "Эти элементы образуют матричную структуру (таблицу)"

### Что теряется
- **Детали содержания:** конкретные числа, слова, имена
- **Контекст:** почему именно эта форма, а не другая
- **Альтернативные интерпретации:** страница может содержать несколько форм одновременно

### Что добавляется
- **Семиотическая типология:** страница отнесена к определённой знаковой форме
- **Стратегия обработки:** для каждой формы — свой экстрактор (L2)
- **Мета-знание:** "это таблица" vs "это текст" — разные способы чтения
- **Готовность к извлечению:** форма определяет, какие структуры искать

## L2: Знаковые формы → Схемы

### Вход
**Знаковая форма** от L1 + **Markdown с координатами** от L0

**Смысл входа:** "Мы знаем форму (например, matrix), но не знаем конкретную структуру"

### Преобразование
**Модуль:** `src/semiotic/extractors.py`

**Деятельность:** Извлечение структурированных схем в зависимости от формы:

| Форма L1 | Экстрактор | Выход |
|----------|-----------|-------|
| topology | `extract_topology()` | Узлы + рёбра графа |
| matrix | `extract_matrix()` | Строки + столбцы + значения |
| hierarchy | `extract_hierarchy()` | Дерево (parent → children) |
| spatial | `extract_spatial()` | Объекты + координаты + отношения |
| enumeration | `extract_enumeration()` | Нумерованный список |
| dynamics | `extract_dynamics()` | Этапы + переходы + условия |
| discursive | `extract_discursive()` | Параграфы + ключевые тезисы |

**Пример для matrix:**
```json
{
  "form": "matrix",
  "columns": ["Показатель", "2023", "2024", "Рост"],
  "rows": [
    {"label": "Выручка", "cells": ["100", "120", "20%"]},
    {"label": "Прибыль", "cells": ["10", "15", "50%"]}
  ]
}
```

**Модель:** qwen3.6:35b (локальная)

### Выход
**Схема** — структурированное представление (JSON):
```json
{
  "page_id": 1,
  "form": "matrix",
  "schema": {
    "columns": [...],
    "rows": [...]
  }
}
```

**Смысл выхода:** "Внутри матричной формы есть конкретная структура: столбцы, строки, значения"

### Что теряется
- **Пространственные координаты:** схема абстрагируется от точного расположения
- **Визуальное оформление:** цвета, шрифты, размеры
- **Контекстные подсказки:** заголовки, подписи, сноски (если не включены в схему)

### Что добавляется
- **Структурная ясность:** явные столбцы, строки, связи
- **Машиночитаемость:** готово для онтологического маппинга
- **Семантическая типизация:** "это заголовок столбца", "это значение ячейки"
- **Готовность к обобщению:** схема может быть сопоставлена с другими схемами

## L3: Схемы → Онтология деятельности (Activity Theory)

### Вход
**Схема** от L2 (JSON)

**Смысл входа:** "Есть структурированные данные, но мы не знаем, какую деятельность они отражают"

### Преобразование
**Модуль:** `src/semiotic/ontology.py`

**Деятельность:** Извлечение структуры деятельности по Activity Theory (Engeström):

1. **Субъект (Subject):** кто действует? (автор, организация, сообщество)
2. **Объект (Object):** на что направлена деятельность? (продукт, услуга, проблема)
3. **Инструменты (Tools):** с помощью чего? (технологии, методы, ресурсы)
4. **Цель (Goal):** зачем? (мотивы, результаты, эффекты)
5. **Сообщество (Community):** кто участвует? (стейкхолдеры, аудитория)
6. **Правила (Rules):** какие ограничения? (нормы, стандарты, барьеры)
7. **Разделение труда (Division of Labor):** кто что делает? (роли, функции)

**Модель:** qwen3.6:35b (локальная)

**Промпт:** `[РОЛЬ] Аналитик деятельности (Activity Theory, Engeström) [ОГРАНИЧЕНИЕ] Только то, что есть в схеме`

### Выход
**Онтология деятельности** (JSON):
```json
{
  "activity": {
    "subject": {"name": "KENGURU.PRO", "type": "организация"},
    "object": {"name": "Цифровые спортивные кластеры", "type": "продукт"},
    "goal": {"description": "Новый стандарт активной жизни"},
    "tools": [],
    "community": [],
    "rules": [],
    "division_of_labor": []
  },
  "relations": [...],
  "model": "Организация KENGURU.PRO формирует цифровые спортивные кластеры..."
}
```

**Смысл выхода:** "Эти структуры отражают деятельность: кто, что делает, с помощью чего, зачем"

### Что теряется
- **Числовые детали:** конкретные значения могут быть агрегированы
- **Формат исходных данных:** JSON vs таблица vs граф — всё становится онтологией деятельности
- **Контекст страницы:** онтология абстрагируется от конкретной страницы

### Что добавляется
- **Структура деятельности:** 7 компонентов Activity Theory
- **Семантика:** "это субъект", "это объект", "это инструмент"
- **Модель деятельности:** краткое описание того, какая деятельность отражена
- **Готовность к анализу:** онтология может использоваться для анализа рисков и возможностей

## L4: Онтология деятельности → Рекомендации

### Вход
**Онтология деятельности** от L3 (activity + relations + model)

**Смысл входа:** "Есть модель деятельности, но мы не знаем, что с ней делать"

### Преобразование
**Модуль:** `src/semiotic/reflector.py`

**Деятельность:** Генерация C-level рекомендаций на основе анализа деятельности:

1. **Стратегическая значимость:** почему это важно для российского бизнеса/государства?
2. **Риски в деятельности:** что может пойти не так? (противоречия между компонентами, слабые связи, барьеры)
3. **Возможности для оптимизации:** где окно для российских экономических операторов?
4. **Рекомендуемое действие:** одно конкретное действие для C-level по оптимизации деятельности
5. **Оценка confidence и urgency:**
   - confidence: насколько уверены в рекомендациях (на основе полноты онтологии)
   - urgency: насколько срочно действовать (на основе рисков)

**Модель:** qwen3.6:35b (локальная)

**Промпт:** `[РОЛЬ] Прагматический рефлектор (Activity Theory) [ОГРАНИЧЕНИЕ] Только из онтологии деятельности, без внешних знаний`

### Выход
**Рекомендации** (JSON):
```json
{
  "page_id": 1,
  "risks": [
    "Рост прибыли (50%) значительно выше роста выручки (20%) — риск устойчивости"
  ],
  "opportunities": [
    "Высокий рост прибыли — возможность масштабирования"
  ],
  "recommended_actions": [
    "Проанализировать структуру затрат",
    "Изучить возможности масштабирования"
  ],
  "confidence": 0.75,
  "urgency": "MEDIUM"
}
```

**Смысл выхода:** "Из доменной модели следуют конкретные риски, возможности, действия"

### Что теряется
- **Детали онтологии:** конкретные сущности и связи абстрагируются в рекомендации
- **Структурная ясность:** рекомендации — это не онтология, а прагматический вывод
- **Прослеживаемость:** если не сохранён provenance, сложно понять, откуда взялась рекомендация

### Что добавляется
- **Прагматическая ценность:** не просто данные, а действия
- **C-level ориентация:** риски, возможности, действия — язык руководителя
- **Confidence и urgency:** мета-информация о надёжности и срочности
- **Готовность к действию:** рекомендации могут быть немедленно использованы

## Итоговая трансформация смысла

| Уровень | Вход | Выход | Смысловой сдвиг |
|---------|------|-------|-----------------|
| **L0** | PDF (физический объект) | Примитивы (текст, вектор, растр) | "Документ" → "Структурированные элементы" |
| **L1** | Примитивы | Знаковая форма | "Элементы" → "Определённая организация" |
| **L2** | Знаковая форма | Схема | "Организация" → "Конкретная структура" |
| **L3** | Схема | Онтология | "Структура" → "Доменная модель" |
| **L4** | Онтология | Рекомендации | "Модель" → "Действия" |

**Ключевой принцип:** На каждом уровне возникает **новый смысл**, которого не было на предыдущем уровне. Это не просто обработка данных — это **деятельность**, порождающая смысл.

## Пример: полный смысловой ход

### Исходный документ
PDF-страница с таблицей финансовых показателей:

```
| Показатель | 2023 | 2024 | Рост |
|------------|------|------|------|
| Выручка    | 100  | 120  | 20%  |
| Прибыль    | 10   | 15   | 50%  |
```

### L0: PDF → Примитивы
```markdown
<|ref|>table<|/ref|><|det|>[[7, 217, 999, 740]]<|/det|>
<table>
  <tr><td>Показатель</td><td>2023</td><td>2024</td><td>Рост</td></tr>
  <tr><td>Выручка</td><td>100</td><td>120</td><td>20%</td></tr>
  <tr><td>Прибыль</td><td>10</td><td>15</td><td>50%</td></tr>
</table>
```

**Смысл:** "Это таблица с координатами"

### L1: Примитивы → Знаковая форма
```json
{"form": "matrix", "confidence": 0.95}
```

**Смысл:** "Это матричная структура (таблица)"

### L2: Знаковая форма → Схема
```json
{
  "columns": ["Показатель", "2023", "2024", "Рост"],
  "rows": [
    {"label": "Выручка", "cells": ["100", "120", "20%"]},
    {"label": "Прибыль", "cells": ["10", "15", "50%"]}
  ]
}
```

**Смысл:** "Внутри таблицы: столбцы (показатель, годы, рост), строки (выручка, прибыль)"

### L3: Схема → Онтология
```json
{
  "entities": [
    {"name": "Выручка", "type": "Финансовый показатель"},
    {"name": "Прибыль", "type": "Финансовый показатель"}
  ],
  "relations": [
    {"from": "Выручка", "to": "Прибыль", "type": "причинно-следственная"}
  ],
  "model": "Финансовая модель: выручка и прибыль"
}
```

**Смысл:** "Это финансовая модель с двумя показателями и причинной связью"

### L4: Онтология → Рекомендации
```json
{
  "risks": ["Рост прибыли (50%) выше роста выручки (20%) — риск устойчивости"],
  "opportunities": ["Высокий рост прибыли — возможность масштабирования"],
  "recommended_actions": ["Проанализировать структуру затрат"]
}
```

**Смысл:** "Из финансовой модели следуют риски и действия"

## Связь с Activity Theory

Каждый уровень конвейера — это **деятельность** с:
- **Субъектом:** модуль (OCR2, classifier, extractor, ontology, reflector)
- **Объектом:** входные данные (PDF, примитивы, форма, схема, онтология)
- **Инструментом:** модель (DeepSeek-OCR2, qwen3.6:35b)
- **Результатом:** выходные данные (примитивы, форма, схема, онтология, рекомендации)
- **Смыслом:** новый смысл, порождаемый деятельностью

**Ключевой вывод:** Система не просто обрабатывает данные — она **порождает смысл** через деятельность на каждом уровне.
