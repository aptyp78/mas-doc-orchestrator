### SMD-соответствие: 9/10

**Сильные стороны:**

1. **Каждая роль — замкнутая единица деятельности.** Объект, инструменты, правила, рефлексия — всё зафиксировано. Это ровно то, что требовалось: «роли должны существовать отдельно от кода».

2. **Граф вместо конвейера.** Модель предложила event-bus архитектуру: «ОРП не вызывают друг друга последовательно. Они подписаны на общий event-bus и реагируют на триггеры.» Это решает primary contradiction (SLA vs deep analysis) через параллелизацию.

3. **L2-рефлекторы анализируют противоречия, а не confidence.** Каждый L2-рефлектор привязан к конкретному противоречию из SMD-карты:
   - Metadata Extractor L2 → ternary (air-gap vs cloud)
   - Disambiguator L2 → secondary (expectations vs LLM quality)
   - Validator L2 → primary (SLA vs depth)
   - Visual Extractor L2 → secondary (expectations vs LLM quality)
   - Graph Builder L2 → primary (SLA vs depth)
   - Dispatcher L2 → синтез всех трёх

4. **Промпты как операционно-ролевые позиции.** Формат `[РОЛЬ] Ты — X. [ОБЪЕКТ] Y. [ИНСТРУМЕНТЫ/ПРАВИЛА] Z.` — это именно заявление роли, а не инструкция «что делать».

**Слабые стороны:**

1. **Избыточная детализация промптов.** Некоторые промпт-шаблоны (особенно Metadata Extractor) слишком длинные — содержат инструкции, которые должны быть в коде, а не в промпте. Роль должна заявлять позицию, а не перечислять шаги.

2. **Dispatcher перегружен.** ОРП 4 (Dispatcher) выполняет функции: динамический порог, таймауты, fallback-маршрутизацию, SLA-контроль, синтез противоречий. Это три роли в одной. Возможно, потребуется разделение на Dispatcher + SLA Controller + Contradiction Synthesizer.

3. **Event-bus не специфицирован.** Модель упоминает «общий event-bus», но не описывает его контракт (формат событий, гарантии доставки, порядок). Это архитектурный риск.

4. **Отсутствует нормализатор.** В конституции ADR-003 не было ОРП для нормализации выхода (Markdown + JSON-sidecar). В текущем коде это делает `pipeline/normalizer.py`. Роль «Normalizer» не спроектирована — она должна быть 7-й ОРП или интегрирована в Graph Builder.

### Противоречия, обнаруженные при валидации

| # | Противоречие | Серьёзность |
|---|-------------|-------------|
| 1 | Dispatcher = 3 роли в одной → риск single point of failure | Высокая |
| 2 | Event-bus упомянут, но не специфицирован → неясен контракт | Средняя |
| 3 | Промпты содержат инструкции, а не только заявление роли → смешение уровней | Средняя |
| 4 | Отсутствует Normalizer ОРП → разрыв в конвейере | Средняя |

## Вердикт

**Результат qwen3.6:35b — принимается с доработками.** Модель корректно применила СМД-подход: спроектировала роли как замкнутые единицы деятельности, связала их в граф (а не конвейер), привязала L2-рефлекторы к противоречиям SMD-карты. Требуются: (1) разделение Dispatcher, (2) упрощение промптов до чистых заявлений роли, (3) спецификация event-bus.
